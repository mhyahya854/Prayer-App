# Prayer App

Cross-platform foundation for a modern Islamic companion app that combines prayer times, Quran reading, duas, worship tracking, and future Google-powered sync features in one product.

## Workspace

- `android`: Android-specific Expo + React Native app workspace
- `ios`: iOS-specific Expo + React Native app workspace
- `web`: Web-specific Expo + React Native app workspace
- `api`: Fastify backend for runtime status, prayer APIs, and future sync/auth flows
- `packages/core`: Shared prayer logic, tracking logic, types, and tests

## What Is Real Now

- Prayer times are calculated locally from saved coordinates, calculation method, madhab, and minute adjustments.
- Prayer tracking is persisted on device and includes current streak, best streak, and recent completion history.
- Local prayer notifications are implemented with per-prayer toggles, shared pre-reminders, rolling native scheduling, bundled sounds, and web push sync support.
- Quran content is locally seeded with offline browsing, search, bookmarks, and reading state.
- Dua content is locally seeded with categories, favorites, and counters.
- Manual location entry and timezone handling are real, including geo-derived timezone lookup and explicit fallback labeling.
- API validation now has a shared error contract, rate limiting, route tests, and web notification sync endpoints.

## Still Partial Or Missing

- Google sign-in, Calendar sync, and Drive backup flows
- Production observability, account auth/linking, and deeper operational hardening
- Production web push still requires deployment-time `DATABASE_URL`, `WEB_PUSH_PUBLIC_KEY`, `WEB_PUSH_PRIVATE_KEY`, `WEB_PUSH_SUBJECT`, `EXPO_PUBLIC_WEB_PUSH_PUBLIC_KEY`, and `ALLOWED_ORIGINS`
- Real production/device notification delivery still requires manual smoke verification with those deployment values in a real browser/device environment
- Fresh installs still emit non-runtime transitive deprecation warnings from the current supported stack: `inflight <- glob@7 <- react-native`, `rimraf@3 <- chromium-edge-launcher <- @react-native/dev-middleware`, and `glob@10.5.0 <- archiver`
- Final approved/licensed launch audio is still missing; the bundled athan/reminder files are starter placeholders and prevent a polished public launch verdict
- Broader content depth such as Quran audio, tafsir, and richer dua metadata

- `npm run preflight`
- `npm run typecheck`
- `npm run test:core`
- `npm run test:api`
- `npm run test:android`
- `npm run test:ios`
- `npm run test:web`

2. Build the canonical clean source artifact:

- `npm run pack:source`

3. Prove the source-only delivery path:

- `npm run verify:fresh`

4. Hand off the generated archive:

- `artifacts/prayer-app-source.zip`

## Clean Validation

`npm run verify:fresh` is the supported end-to-end handoff proof. It creates `artifacts/prayer-app-source.zip`, extracts that clean archive into a temporary directory, runs `npm ci`, then runs:

- `npm run typecheck`
- `npm run test:core`
- `npm run test:api`
- `npm run test:android`
- `npm run test:ios`
- `npm run test:web`

The supported artifact is the source-only archive generated by `npm run pack:source`, not a manual zip of the working tree.

## Production Notification Deployment

Before running production-like smoke checks locally, copy `.env.example` to `.env` and provide real deployment values.

For the notification stack, keep these values aligned across the API, web client, and deployment platform:

- `DATABASE_URL`: required for durable web push subscriptions, saved notification profiles, and queued delivery jobs
- `WEB_PUSH_PUBLIC_KEY`: required on the API for VAPID identity and must match the client public key
- `WEB_PUSH_PRIVATE_KEY`: required on the API to sign web push deliveries
- `WEB_PUSH_SUBJECT`: required on the API as the VAPID contact identity for production delivery
- `EXPO_PUBLIC_WEB_PUSH_PUBLIC_KEY`: required in the web client to create browser push subscriptions; must match `WEB_PUSH_PUBLIC_KEY`
- `ALLOWED_ORIGINS`: required on the API in production so the web app can call `/api/notifications/web/*`

If any of those values are missing, production web push will not be fully functional.

For deployment-day gating, run:

- `npm run preflight:release`

`preflight:release` fails if required runtime env values are missing or if launch audio is still marked as placeholder/not approved in `assets/sounds/ASSET_LICENSES.md`.
